# Scroll Down Sports iOS Screen Support Braindump

## Goal

Make Scroll Down Sports feel intentional on every supported iOS screen: small iPhones, large iPhones, landscape phones, iPads, iPad Split View, Stage Manager windows, Dynamic Type, and dark mode.

This is not a request for a separate iPad app. The target is one SwiftUI app with adaptive layout rules, deterministic test coverage, and no stretched phone UI on large screens.

## Current State

- The app is SwiftUI, generated by XcodeGen from `project.yml`.
- Deployment target is iOS 18.0.
- The main shell is `NavigationStack` in `ContentView`.
- `HomeView` and `GameDetailView` are mostly single-column scroll views with fixed horizontal padding.
- Snapshot coverage currently models phone widths: small, compact, large, and phone landscape.
- `Scripts/local_gate.sh` uses iPhone 17 Pro on iOS 26.2 as the canonical simulator and the visual gate intentionally pins to that runtime.
- CI has iPhone visual, UI, accessibility, and performance gates, but no explicit iPad or multitasking gate.

The main risk is not "the app crashes on iPad." The risk is that it technically runs while reading as a blown-up phone layout with long lines, oversized empty space, weak navigation ergonomics, and untested split-window behavior.

## Hard Rules

- Support every iOS size class with one product model, not device-specific forks.
- Avoid fixed full-screen assumptions. Screens must react to available width, safe areas, orientation, Dynamic Type, and keyboard presence.
- Do not stretch readable content edge to edge on iPad. Use max widths, columns, or split navigation.
- Do not hide core actions in iPad-only chrome. Compact and regular layouts should expose the same product capabilities.
- Use system navigation, sheets, popovers, menus, safe-area handling, and Dynamic Type before custom layout machinery.
- Keep the catch-up product simple. iPad support should improve scanning and reading, not reintroduce dashboard clutter.
- Every adaptive decision needs a test or snapshot that would fail if someone regresses it.

## Target Device Matrix

Minimum manual and automated coverage should include:

- Small iPhone portrait: iPhone SE class, compact width.
- Modern iPhone portrait: iPhone 17 Pro class, compact width.
- Large iPhone portrait: Pro Max class, compact width.
- iPhone landscape: compact height, wider content, toolbar pressure.
- iPad portrait full screen: regular width and regular height.
- iPad landscape full screen: regular width, wide reading surface.
- iPad Split View narrow: compact-width iPad window, should behave like a phone without clipping.
- iPad Split View medium: transition zone where content should not feel stranded.
- Stage Manager or resizable iPad window: arbitrary width changes must not break sticky bars, sheets, or scroll restoration.
- Dynamic Type accessibility sizes on at least one compact and one regular-width layout.
- Dark mode on at least one compact and one regular-width layout.

## Layout Model

Create an explicit adaptive layout layer instead of scattering width checks through views.

Recommended shape:

```swift
struct AppLayoutMetrics {
    let horizontalPadding: CGFloat
    let contentMaxWidth: CGFloat?
    let cardGridColumns: [GridItem]
    let detailMaxWidth: CGFloat
    let usesSplitNavigation: Bool
}
```

The metrics should be derived from environment and geometry:

- `horizontalSizeClass`
- `verticalSizeClass`
- available container width
- Dynamic Type size
- idiom only as a weak hint when size classes and actual width are not enough

Use actual available width as the final source of truth. iPad multitasking can make an iPad window phone-sized, and a large landscape phone can create width pressure that does not map cleanly to device family.

## Navigation

Compact width:

- Keep the current `NavigationStack` feel.
- Home opens game detail by pushing onto the stack.
- Sticky top and bottom detail controls remain inside the detail screen.

Regular width:

- Move toward `NavigationSplitView`.
- Sidebar/content column: home feed, filters, pinned games, timeline sections.
- Detail column: selected game catch-up stream.
- Preserve the empty-detail state when no game is selected.
- Preserve deep links and direct detail entry.

Implementation preference:

- Introduce an app shell that chooses compact stack vs regular split.
- Keep `HomeView` and `GameDetailView` reusable rather than creating `HomeViewForIPad`.
- Selection state should live in the shell, not inside rows.

## Home Screen

Compact:

- Single-column feed.
- Sticky filter header remains concise.
- Existing card density and controls are acceptable as the baseline.

Regular:

- Do not stretch every row to full iPad width.
- Prefer one of these patterns:
  - centered readable feed with max width around 720 to 860 points, or
  - adaptive two-column game grid when there are enough games and the card design supports scanning.
- Filters can become a compact toolbar/sidebar group, but should not consume a huge vertical band.
- Pinned games should stay visually distinct without becoming a second dashboard.
- Empty, loading, and error states should use the same max-width policy as content.

Regression risks to test:

- segmented league picker overflow
- team search field becoming too wide
- section header subtitle wrapping badly
- pin button drifting away from its card content
- card rows stretching into unreadable line lengths

## Game Detail Screen

Compact:

- Keep the current reader flow: header, resume, controls, play stream, stats, scoreboard.
- Bottom new-play affordance remains reachable and does not cover content.

Regular:

- The play stream should be a readable column, not a full-width wall of text.
- Target detail content width should usually sit around 680 to 820 points.
- The scoreboard and stats can use wider layouts inside that column, but individual text rows still need readable line lengths.
- Sticky navigation should align with the content column, not the whole screen edge.
- Live-follow and resume affordances must remain visible when the user resizes an iPad window.
- If a side rail is added later, it should carry secondary state only: score context, pinned status, or quick anchors. It must not duplicate the main play stream.

Regression risks to test:

- scroll anchors restored to the wrong event after width changes
- bottom safe-area affordance covering the scoreboard
- sticky top bar spanning awkwardly on iPad
- raw play text lines becoming too long
- stats tables either clipping or becoming sparse
- confirmation dialogs presenting as awkward full-width sheets instead of appropriate popovers/dialogs

## Modals, Menus, And Controls

- Prefer `confirmationDialog`, `contextMenu`, `Menu`, and `popover` semantics that adapt automatically.
- Avoid custom full-screen overlays for small interactions.
- Any iPad popover needs an obvious source control and should survive rotation/window resizing.
- Minimum tap target remains 44 by 44 points.
- Toolbar actions should use symbols and accessibility labels.
- Do not add separate explanatory text for iPad. The layout should make the behavior obvious.

## Safe Areas And Insets

- Keep top and bottom bars inside `safeAreaInset` where they are part of screen chrome.
- Avoid `ignoresSafeArea` except for intentional backgrounds.
- Test home and detail with:
  - portrait
  - landscape
  - iPad full screen
  - iPad Split View narrow
  - Dynamic Type accessibility sizes
- Bottom affordances must add enough content padding or inset spacing that the last row and scoreboard can scroll fully above them.

## Dynamic Type

Dynamic Type support is part of screen support, not a separate accessibility task.

Rules:

- Text can wrap; controls cannot overlap.
- Controls that become too dense should stack vertically at accessibility sizes.
- Avoid hard-coded heights for text containers.
- Fixed icon sizes are fine, but labels need flexible width.
- Snapshot at `.accessibility2` for compact and regular-width layouts.

High-risk current surfaces:

- segmented league picker
- home section headers with title plus subtitle
- game rows with teams, metadata, status, and pin affordance
- stream control bar
- resume banner
- sticky detail navigation bar
- stat tables

## Testing Plan

### Snapshot Support

Extend `SnapshotDevice` and `SnapshotWidth` to include iPad and multitasking sizes.

Suggested additions:

- `ipadPortrait`: 834 by 1194
- `ipadLandscape`: 1194 by 834
- `ipadLargeLandscape`: 1366 by 1024
- `ipadSplitNarrow`: 504 by 1112
- `ipadSplitMedium`: 744 by 1112

Suggested widths:

- `phoneCompact`: existing 343 to 393 content widths
- `tabletReadable`: about 720
- `tabletWide`: about 860
- `splitNarrow`: about 472

Do not snapshot every state on every device. Pick states that prove layout rules:

- Home feed, no pinned games, iPad portrait.
- Home feed, pinned games, iPad landscape.
- Home with filters active, iPad Split View narrow.
- Home with Dynamic Type accessibility, iPad portrait.
- Detail top chrome and controls, iPad portrait.
- Detail play stream, iPad landscape.
- Detail stats and scoreboard, iPad landscape.
- Detail empty play-by-play, iPad Split View narrow.
- Detail dark mode, iPad portrait or landscape.

### UI Tests

Add at least one iPad UI smoke route:

1. Launch on iPad simulator.
2. Verify home loads.
3. Select a game.
4. Verify detail appears.
5. Jump latest/end.
6. Return or select another game without losing navigation state.

For regular-width split navigation, add a UI assertion that home and detail can coexist when the layout is in split mode.

### Accessibility

Run accessibility audits on both:

- compact iPhone destination
- iPad destination

Specific checks:

- no clipped labels in large text
- all icon-only controls have labels
- no duplicate confusing controls in split mode
- filter controls remain reachable
- detail sticky controls remain reachable

### CI

Keep the current iPhone canonical visual gate for baseline stability.

Add separate iPad gates rather than replacing the phone gate:

- `ipad-visual`: pinned iPad simulator and selected iPad snapshots.
- `ipad-ui-smoke`: iPad simulator critical flow.
- `ipad-accessibility`: iPad accessibility audit.
- `multitasking-layout`: snapshot or UI route at iPad split widths.

The current `TEST_DESTINATION` override in `Scripts/local_gate.sh` is a good starting point, but the visual gate currently unsets it. That is correct for the existing phone baseline, but an iPad-specific visual gate should choose its own pinned destination instead of sharing the phone-only visual gate.

## Implementation Order

1. Add layout metrics and device/width snapshot support.
2. Add failing iPad and split-width snapshots for the current layout.
3. Introduce an adaptive app shell with `NavigationSplitView` for regular width.
4. Constrain home and detail content widths so large screens stop stretching phone rows.
5. Adapt home scanning for regular width: centered feed first, optional grid only if it improves real use.
6. Align sticky detail bars and bottom affordances to the readable content column.
7. Fix Dynamic Type pressure in headers, controls, resume, and stats.
8. Add iPad UI smoke and accessibility gates.
9. Add CI jobs for iPad support after local gates are stable.
10. Record new snapshot baselines intentionally with `SNAPSHOT_RECORD=1`.

## Definition Of Done

iPad and all-screen support is done when:

- Compact iPhone layout still passes existing gates.
- iPhone landscape has no clipped toolbar/header/control content.
- iPad full-screen portrait and landscape do not look like stretched phone screens.
- iPad Split View narrow behaves like a compact app without clipping.
- Regular-width navigation supports home/detail ergonomically.
- Dynamic Type accessibility sizes do not overlap or hide primary actions.
- Sticky top and bottom detail chrome align with content and never cover final content.
- Snapshot, UI smoke, and accessibility gates include at least one iPad destination.
- CI runs the iPad gates separately from the canonical phone visual baseline.
- Manual simulator verification has been done on the newest installed iPad runtime.

## Explicit Non-Goals

- No macOS/Catalyst work unless requested separately.
- No separate iPad-only feature set.
- No dashboard rewrite.
- No new sports data model work.
- No backend changes.
- No visual redesign just for tablet scale.

## Open Decisions

- Should regular-width home use a centered single-column feed first, or an adaptive two-column grid?
- Should regular-width app shell immediately adopt `NavigationSplitView`, or should content-width constraints land first?
- Which iPad simulator should be canonical for local iPad visual baselines?
- Should split-width snapshots live in unit snapshot tests only, or should CI also run XCUITest on an iPad Split View style destination?
- Should the detail screen eventually gain a secondary side rail, or should it remain a focused single reading column on all devices?

## First Practical Pass

The smallest useful first PR:

1. Add iPad sizes to snapshot support.
2. Add one home and one detail iPad snapshot that fail under the current stretched layout.
3. Add `AppLayoutMetrics`.
4. Center and max-width home/detail content on regular-width screens.
5. Add a focused iPad visual gate or dry-run script contract.
6. Run:

```sh
Scripts/local_gate.sh visual
TEST_DESTINATION='platform=iOS Simulator,name=iPad Pro 11-inch (M5),OS=26.2' Scripts/local_gate.sh ui-smoke
TEST_DESTINATION='platform=iOS Simulator,name=iPad Pro 11-inch (M5),OS=26.2' Scripts/local_gate.sh accessibility
```

After that pass, decide whether `NavigationSplitView` should be the next step or whether content constraints are enough for the first iPad-supported release.
